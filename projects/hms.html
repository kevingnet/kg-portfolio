int InputValidatorPCRE::GetPCRECompileOptions( const string& options )
{
	int PCRECompileOptions = 0;
	if ( options.size() == 0) return 0;
	for( int i=0; i<PCRE_OPTION_COUNT; i++ ) {
		if(options.find(sm_PCRECompilationOptions[i].option) != string::npos)
			PCRECompileOptions |= sm_PCRECompilationOptions[i].PCREOption;
	}
	return PCRECompileOptions;
}

In fact, it is estimated that over 90 percent of them relate to lack or poor input validation. Therefore, an Input Validation Library was developed to make a programmers life easier. This document will attempt to describe in detail what this libray does, how it does it, what problems it solves and what its limitations are. 2 Scope, methodologies and limitations 2.1 Scope The Input Validation Library provides development teams and programmers with input validation capabilities by checking existence of input, length of input, value range checking for standard data types, single character validation for strings and PCRE (Perl Compatible Regular Expressions) checks. 2.2 Methodologies The library was designed to work with C/C++ and Perl programs, however its use could be extended to any other systems, since those technologies are widely supported. The library was implemented in C++ using Object Oriented Methodologies. It's interface was made compatible with C so that any C application can use it by simply including its header file and linking with the library. Additionally, its interface was made available to Perl by compiling a wrapper that can be invoked by Perl the same way Perl would invoke any standard module. 2.3 Limitations The library has only been tested as a unique running instance per computer, therefore for the moment it can only be used on one application per server. More testing will be required in order to find out if it can run multiple instances. When the library validates input and the validation does not pass the test, this probably indicates some kind of tampering or hacking attempt. By design, the library does not know “who” is doing that, who is trying to hack or at least who is sending bad data. Therefore, it is the application's responsibility to perhaps make a note of that or act upon that information. Perhaps, a desirable “feature” would be to allow an application to pass this “extra” information to the library, so that the library can handle this in a “standard” way. At the moment the library only saves this information in a stats file. 3 Application description 3.1 What does this library do? The following are the steps the library takes during input validation: NOTE: “item” refers to a type of validation, “string” refers to the input to be validated. Ensure we have a string and that it is not empty Compare string size with global minimum and maximum requirements Compare string size with item's minimum and maximum requirements If item is a standard data type ensure it conforms to proper type and range If item is a standard data type and has a value range ensure string is within proper range If item is a string ensure all non alpha-numeric characters are valid If item is a string ensure any repeating non alpha-numeric characters are valid If item has a regular expression validate the string using the regular expression If at any point a validation step does not pass, all further tests are terminated for that string. A more detailed explanation is provided in section 6.2. 3.2 Functionality The Input Validation Libray uses a binary file as its configuration wich is compiled with a utility that the Enterprise Security Team developed. Let's discuss this utility first since it's the first part of this library. The utility is called PCREBinGen and it's an executable file developed in C++. When PCREBinGen is run, it looks for a file named PCREBinGen.xml unless another name is given at the command line. A number of other parameters are also accepted as optional and those deal with settings that control how the XML file is to be processed. When PCREBinGen.xml is read, it is parsed and the data is processed internally to produce an internal document; this document contains global variables that were found in the xml file. If some settings were missing it will use its internal defaults. The xml file must also contain any number of entries named ValidateItem. A ValidateItem entry must contain settings that control how the input validation library is to operate on the data that is to be validated. The format of the xml file is provided in Section 6 – Internals. When the parsing is done PCREBinGen has an object which contains all global variables and all the ValidateItem entries it found. This utility will then process each ValidateItem and will invoke PCRE Library which it uses to compile any PCRE string found. The data produced by the PCRE compile function is appended to the ValidateItem object contained in the InputValidationDocument. When the utility finishes processing all items it creates a file called libinval.pcre. At this point PCREBinGen's work concludes. The name of the library is libinval.so which stands for, 'lib' the standard unix library prefix, 'in' for input, and 'val' for validation. libinval.pcre is used by the Input Validation Library as its configuration file. The main reason that the data is compiled separately by PCREBinGen is to gain performance. What this means is that the PCRE expression is compiled only once and the data is saved for future use. Other systems compile the data each time a check is performed, this of course takes time, and although the time it takes is negligible when you add it all up, it can be a considerable length of time. As a side effect the library configuration file is kept in binary format this appears to make the system more secure. However, is worth mentioning that in this version, although the data in the file is binary, not all of it is unreadable. Therefore, anyone with a binary file viewer can figure out some interesting things about the file. The library could be improved in the next release by including a hashing and check-sum function which would make the file unreadable. At this point we have two files that comprise our Input Validation Library, the library itself and libinval.pcre. Usage information is demonstrated in the next chapter and configuration data is provided in the following chapters. Once we have an application to use the library. Validation can be done for the following data types: (signed/unsigned) char, short int, int, long, long long; float, double, bool and of course zero terminated strings. Note: the library validates strings only, this means that standard data type validation is done first by converting the string. As a side effect from validating standard data types the library can also provide the converted data. However, when a string is validated for compliance with a standard data type the validated value can be returned as the standard data type value. The library keeps counters on all validations performed, total, passed, failed, errors and other counters that are more specific. This information is saved to a file upon library exit. Developers that use the library are encouraged to save the counters to a file periodically, this is so that in the event of an application or system failure we have statistical information that will prove useful later. 3.3 Technologies Gnu's g++ was used to compile the utility and library. The PCRE Library from www.pcre.org was used to perform regular expression checks. This library is widely used by a large number of companies including HMS, google, etc... libinval.so is linked statically to PCRE library, therefore it is unecessary to keep a local copy on the machine where the library is to be used. This decreases dependencies and facilitates the installation and use of the library. In other words developers do not need to install pcre, libinval has already been linked statically to it. Of course a disadvantage is that when pcre provides upgrades they will not be used by the library. However, the library can be provided with dynamic linkage, in that case it will benefit from pcre upgrades. If the library is linked dynamically and the pcre compiled data binary format should change, it may be necessary to reproduce the libinval.pcre file to adopt the new format. PCRE is quite ubiquitous and offers very high performance. The recommended way of dealing with input as you may already know is not by filtering “invalid” characters, but by discarding everything that is not considered “valid.” This presents some problems if we were to compose regular expressions in this manner. The regular expressions would become necessary large or at least more complex, therefore affecting performance. We have addressed this issue by coding another mechanism that deals with discarding invalid characters. In the xml configuration file, the security or sysadmin will have to type all “valid” characters in the globals section. This section will then be applied to all items in the xml configuration that do not have a “valid” characters section of its own. Additionally, there is another section that deals with “valid” repeatable characters. These are characters that can appear in a string contiguously. By design, all alpha-numeric characters (a-z 0-9) space and new line characters can be repeated. In the case of space and new line characters that is true only if you allow space and multiline characters in the configuration file, for that item. Therefore, if you expect any other repeatable characters to appear in a string, you must make sure they are added to the configuration file. There are a few other things that are adjustable through the configuration file. Such as character case sensitivity. If you set an item to be case insensitive, its PCRE data will be compiled with that option, that should also speed things up. The Apache project XERCES XML parser library was used for PCREBinGen only, the library does not deal with XML at all in order to keep a small footprint. A logging system similar to the one used in the neti project from HMS was used for output of information and other messages from the library. 4 Usage The system was designed to be simple to use. Although the library is relatively easy to use, what takes the most time is editing the PCREBinGen.xml configuration file used to produce the binary file. This must be done very carefully and tested thoroughly to insure proper input validation. Fortunately, the Enterprise Security Team would produce prepackaged ValidateItems that would be useful in most situations and applications, and only those times deemed necessary should we produce new entries. The xml validation file should not contain a comprehensive list of all the ValidateItem entries. Each application should have its own configuration file. This would make the library more efficient in its usage of memory, so that only the items that are necessary to validate on a per application basis would be kept in RAM. 4.1 Initial Setup and configuration Before anyone attempts to use the library, the teams must agree on what's going to be validated and craft the necessary regular expressions carefully and test them thoroughly. Failure to do so may result in either a possible vulnerability or denial of service. After this step is completed, the Enterprise Security Team will prepare a PCREBinGen.xml file with the necessary information, then a libinval.pcre will be generated and sent to the development team. Initially there should be 3 files that are necessary; libinval.so: This is the library file and it must be found in either of these locations; /usr/lib or /usr/local/lib. Alternatively, the file can be copied anywhere and a symbolic link pointed to it can be placed in any of the above mentioned locations. libinval.pcre: This is the configuration file and it must be found in either of these two locations; /etc/libinval/libinval.pcre or /usr/local/etc/libinval.pcre The other file is libinval.h header file for C/C++ projects or InVal.pm Perl Module for Perl projects. Actually, in Perl there are other files that must be copied in order to use InVal.pm module which is a wrapper. Alternatively, Perl programmers can use inline C embeded in perl code, which may improve performance and does not necessitate the extra Perl files. At this point, the development team must know what is contained in the binary file, that is, the global defaults and each of the ValidateItem entries. Each of this entry has a name and that is how the library identifies each entry which is a type of validation. In order to increase the library's performance, a different way of addressing the entries was devised, instead of using its name. Each entry is addressed by an ID. This ID is not known until runtime, (well, it is really the numerical order of each of the entries in the XML file, but this may change in future versions and must not be relied upon,) therefore, the programmer must obtain each entry ID based on its name. The names are case sensitive and can be things like Email, Name, Age, PhoneNumber, etc... The programmer must keep a reference to this ID throughout the runtime of their application. These Ids must be obtained each time the application is started. Although, this may seem like an extra step, it is a necessary one in order to avoid calculating this internally each time by string comparison against possibly all entries. This, of course, was done in order to increase run time performance. PERFORMANCE TIP: Put items that are to be validated the most in the beginning of the xml file. This will save a few cpu ticks. The input validation library outputs information to STDOUT and STDERR and by default will output at a logging level of LOG_INFO (please see next chapter for meaning.) Programmers may do as they please with this information, i.e. redirect to system log or capture output. The library tries to write statistic information to a file /var/log/libinval.stats, make sure that the library or program has permissions to write to that location. Otherwise the stats will be saved in /var/tmp/libinval.stats The stats file contains a header with enough information to understand the format. The content is text. The function iv_save_stats() must be called periodically, otherwise stats may be lost. Stats information is crucial in order for the teams to understand what's been going on during input validation and to determine if adjustments to the configuration file will be necessary or to act upon the information learned in other ways. 4.2 Usage of libinval.so in C/C++ projects Project preparation: Add: #include "libinval.h" to the project or file that would call library routines. Program initialization: Call: idItem = iv_get_validate_item_id( “ValidateItem name” ); For each item. Let's keep in mind that at this point the programmer knows what types of validation exist and knows them by name. If returned value is -1 an error occurred or the item was not found. Input Validation: Call: result = iv_validate_item( szStringToValidate, idItem ); Result: 1 = Validation Passed 0 = Validation Failed -1 = An error condition was encountered 4.3 Usage of libinval.so in Perl projects The syntax would be similar to the one above. Perl programmers can use inline C or can use InVal.pm which is a wrapper. When using the wrapper, there are other files which must be copied into either the Perl system library/module directory or a user local Perl directory. The extra files are: Inval.pm in: /usr/lib/perl5/site_perl/5.8.6/i686-linux/ and: InVal.bs autosplit.ix Inval.so in: /usr/lib/perl5/site_perl/5.8.6/i686-linux/auto/InVal Note: in our tests, when using the wrapper, there was a time increase of about 95% than when using C. We have not tested inline C in Perl yet but it is presumably faster. 5 API From libinval.h //return values #define IV_VALIDATION_PASSED 1 #define IV_VALIDATION_FAILED 0 #define IV_VALIDATION_ERROR -1 //log options enum { LOG_FATAL, //0 LOG_CRIT, //1 LOG_ERR, //2 * default LOG_WARN, //3 LOG_NOTICE, //4 LOG_INFO, //5 LOG_DEBUG //6 }; Depending on the logging level selected, the library will output anything equal to or less than the level selected. The following is a list of the main classes and their respective explanation: PCREBinGenHandlers Derived from xerces xml parser handler class. Handles parsing and has functions that are called by the xerces system to notify the application and provide data. It uses other classes from xerces and it has the class object InputValidatorXMLReader, which it uses as a temporary buffer to save data from the xml file PCREBinGen.xml or the one selected by the user. InputValidatorXMLReader When the parser finishes reading the xml file this class will have the xml data in a InputValidatorDocument object. So, the only purpose of this class is to intercept the data the parser throws at us and to build the InputValidatorDocument object. This class also has a set of defaults that can and will be used should the xml file be found incomplete, to a point. InputValidatorDocument This class is the most important one for both the utility and the library. It contains the globals and all ValidateItem entries including any invalid entries in class objects InputValidatorValidateItem in a stl vector. This class has the capability so serialize itself, that is, given a file name (hopefully, libinval.pcre,) it will load the file and build itself into existence complete with globals and ValidateItem entries. It can also save itself, (really, no pun intended ) to another file. The class is versioned and it stores PCRE library version information in the binary file. InputValidatorValidateItem These objects are managed by InputValidatorDocument and get their content from the xml file ValidateItem entries. InputValidatorValidateItems actually do the input validation. They has data type knowledge and since all data is private, it has public accessor methods. These objects keep binary data that is produced by the PCRE library's pcre_compile2 and pcre_study. The actual data compilation is orchestrated by PCRECompiler. PCRECompiler This class iterates through all InputValidatorValidateItem found in InputValidatorDocument m_sPCRE class member data and calls the functions mentioned above. If the function call pcre_compile2 succeeds information on the compile data is obtained such as its length. The data and information is saved in the InputValidatorValidateItem. Then this object calls pcre_study which is a function that will return extra data that will cause the input validation to be faster in some circumstances. If the study function returns any additional data, information about it is obtained and the length and extra data are saved. BinSerializer This class is used by PCREBinGen to save and load files. StdDataTypes This class is used to handle standard data types validation and conversions. It is a wrapper for standard C functions. ErrorHandler This class is used to obtain system and exception error information and to process it in a standard and modular way. It also serves as a buffer for errors, so that later they can be dealt with accordingly. 6.1 PCREBinGen This utility started from one of the samples included in the Xerces samples directory to save a lot of time. It was built from SAXPrint. SAXPrint will take a xml file and other parameters from the command line, will parse the file and output to STDOUT. 				{
					m_bufRepeats[m_RepeatsIndex++] = m_c;

				}
			} 
			else

#endif
				return IV_VALIDATION_FAILED;

		} 
	}
	//check allowed repeating characters
	for( m_i=0; m_i<m_RepeatsIndex; m_i++ ) 
	{
		if( m_bufRepeats[m_i] == 0 ) break;
		if( strchar2(static_cast<const char*>(m_pAllowedRepeatingChars), m_bufRepeats[m_i], m_AllowedRepeatingCharsLength) == 0 )
		{
#ifdef ProcessStats
			StatsValidationsFailed_InvalidRepeatChar++;
#endif
			m_LastValidationFailure = VF_InvalidRepeatChar;
			return IV_VALIDATION_FAILED;
		}
	}
	return IV_VALIDATION_PASSED;
}

This document will attempt to describe in detail what this libray does, how it does it, what problems it solves and what its limitations are. 2 Scope, methodologies and limitations 2.1 Scope The Input Validation Library provides development teams and programmers with input validation capabilities by checking existence of input, length of input, value range checking for standard data types, single character validation for strings and PCRE (Perl Compatible Regular Expressions) checks. 2.2 Methodologies The library was designed to work with C/C++ and Perl programs, however its use could be extended to any other systems, since those technologies are widely supported. The library was implemented in C++ using Object Oriented Methodologies. It's interface was made compatible with C so that any C application can use it by simply including its header file and linking with the library. Additionally, its interface was made available to Perl by compiling a wrapper that can be invoked by Perl the same way Perl would invoke any standard module. 2.3 Limitations The library has only been tested as a unique running instance per computer, therefore for the moment it can only be used on one application per server. More testing will be required in order to find out if it can run multiple instances. When the library validates input and the validation does not pass the test, this probably indicates some kind of tampering or hacking attempt. By design, the library does not know “who” is doing that, who is trying to hack or at least who is sending bad data. Therefore, it is the application's responsibility to perhaps make a note of that or act upon that information. Perhaps, a desirable “feature” would be to allow an application to pass this “extra” information to the library, so that the library can handle this in a “standard” way. At the moment the library only saves this information in a stats file. 3 Application description 3.1 What does this library do? The following are the steps the library takes during input validation: NOTE: “item” refers to a type of validation, “string” refers to the input to be validated. Ensure we have a string and that it is not empty Compare string size with global minimum and maximum requirements Compare string size with item's minimum and maximum requirements If item is a standard data type ensure it conforms to proper type and range If item is a standard data type and has a value range ensure string is within proper range If item is a string ensure all non alpha-numeric characters are valid If item is a string ensure any repeating non alpha-numeric characters are valid If item has a regular expression validate the string using the regular expression If at any point a validation step does not pass, all further tests are terminated for that string. A more detailed explanation is provided in section 6.2. 3.2 Functionality The Input Validation Libray uses a binary file as its configuration wich is compiled with a utility that the Enterprise Security Team developed. Let's discuss this utility first since it's the first part of this library. The utility is called PCREBinGen and it's an executable file developed in C++. When PCREBinGen is run, it looks for a file named PCREBinGen.xml unless another name is given at the command line. A number of other parameters are also accepted as optional and those deal with settings that control how the XML file is to be processed. When PCREBinGen.xml is read, it is parsed and the data is processed internally to produce an internal document; this document contains global variables that were found in the xml file. If some settings were missing it will use its internal defaults. The xml file must also contain any number of entries named ValidateItem. A ValidateItem entry must contain settings that control how the input validation library is to operate on the data that is to be validated. The format of the xml file is provided in Section 6 – Internals. When the parsing is done PCREBinGen has an object which contains all global variables and all the ValidateItem entries it found. This utility will then process each ValidateItem and will invoke PCRE Library which it uses to compile any PCRE string found. The data produced by the PCRE compile function is appended to the ValidateItem object contained in the InputValidationDocument. When the utility finishes processing all items it creates a file called libinval.pcre. At this point PCREBinGen's work concludes. The name of the library is libinval.so which stands for, 'lib' the standard unix library prefix, 'in' for input, and 'val' for validation. libinval.pcre is used by the Input Validation Library as its configuration file. The main reason that the data is compiled separately by PCREBinGen is to gain performance. What this means is that the PCRE expression is compiled only once and the data is saved for future use. Other systems compile the data each time a check is performed, this of course takes time, and although the time it takes is negligible when you add it all up, it can be a considerable length of time. As a side effect the library configuration file is kept in binary format this appears to make the system more secure. However, is worth mentioning that in this version, although the data in the file is binary, not all of it is unreadable. Therefore, anyone with a binary file viewer can figure out some interesting things about the file. The library could be improved in the next release by including a hashing and check-sum function which would make the file unreadable. At this point we have two files that comprise our Input Validation Library, the library itself and libinval.pcre. Usage information is demonstrated in the next chapter and configuration data is provided in the following chapters. Once we have an application to use the library. Validation can be done for the following data types: (signed/unsigned) char, short int, int, long, long long; float, double, bool and of course zero terminated strings. Note: the library validates strings only, this means that standard data type validation is done first by converting the string. As a side effect from validating standard data types the library can also provide the converted data. However, when a string is validated for compliance with a standard data type the validated value can be returned as the standard data type value. The library keeps counters on all validations performed, total, passed, failed, errors and other counters that are more specific. This information is saved to a file upon library exit. Developers that use the library are encouraged to save the counters to a file periodically, this is so that in the event of an application or system failure we have statistical information that will prove useful later. 3.3 Technologies Gnu's g++ was used to compile the utility and library. The PCRE Library from www.pcre.org was used to perform regular expression checks. This library is widely used by a large number of companies including google, etc... libinval.so is linked statically to PCRE library, therefore it is unecessary to keep a local copy on the machine where the library is to be used. This decreases dependencies and facilitates the installation and use of the library. In other words developers do not need to install pcre, libinval has already been linked statically to it. Of course a disadvantage is that when pcre provides upgrades they will not be used by the library. The recommended way of dealing with input as you may already know is not by filtering “invalid” characters, but by discarding everything that is not considered “valid.” This presents some problems if we were to compose regular expressions in this manner. We have addressed this issue by coding another mechanism that deals with discarding invalid characters. In the xml configuration file, the security or sysadmin will have to type all “valid” characters in the globals section. This section will then be applied to all items in the xml configuration that do not have a “valid” characters section of its own. Additionally, there is another section that deals with “valid” repeatable characters. These are characters that can appear in a string contiguously. By design, all alpha-numeric characters (a-z 0-9) space and new line characters can be repeated. The library serves as a wrapper for PCRE ( www.pcre.org ) written by Philip Hazel , which appears to have done an excellent job in developing it. It is used and many well known applications such as Exim MTA, Python, Apache, PHP, KDE, Postfix, Analog, and many others. Google uses it as well and they developed a C++ wrapper for it. Libinval.so is more than a wrapper because of the extra checks it performs. Those extra checks allow us to write much simpler regular expressions that will perform presumably much faster than doing it all through PCRE. The use of this library facilitates programming, since the responsibility for input validation is delegated to the library for the most part. It serves as a way to organize and consolidate knowledge which can be shared company wide by virtue of using standardized validations, which are encapsulated in the binary configuration files. The data that the library produces in the stats file can also prove quite useful. This data can be imported into databases for statistical analysis. This is useful because it would allow middle and upper management to make decisions based on the knowledge gained from statistical analysis. If, for example, you have lots of validation failures due to “invalid extra characters,” what's going on here? At this point management directs systems and/or security departments to investigate and possibly obtain logs or other forensic information that could uncover the “identity” or at least the origin of the “bad” data. At that point it can be decided what can be done to remedy or mitigate this situation. Another important factor to note is for example, an application is programmed to use the library and a vulnerability is later encountered. The vulnerability can be handled by editing the configuration and recompiling the binary file. Even in the extreme case, should the library have to be modified, it can be done so, without necessarily affecting the main application. It is also necessary to say that this library will not be a cure for all, be everything for everybody, security all in one solution. It is the developers responsibility to code securely and the Enterprise Security Team to help doing so and to analyze code and applications for other types of vulnerabilities. The library was designed to validate input using PCRE and be as easy to use, inobstrusive and fast as possible. We believe that it has met those requirements and our team highly recommeds its use and inclusion into all applications company wide as soon as possible.

Let's discuss this utility first since it's the first part of this library. The utility is called PCREBinGen and it's an executable file developed in C++. As it is well understood the web is very susceptible to attacks. One way to protect against some types of attacks is through using Intrusion Detection Systems. Snort from www.snort.org provides software that protects systems using IDS methods. Snort works by having a set of signatures and applies them to incoming network traffic. Zeus has adopted some of that functionality by using snort rule sets that apply to the web and incorporates them into their web server software. Our team is looking into how HMS can take advantage of that technology and incorporate it into all Zeus web servers. In the following sections we will attempt to explain the benefits of adopting this technology, its limitations, and implementation details. We will also examine our current web server vulnerabilities and what types of vulnerabilities would be mitigated by using zeus-snort rule sets. 2 Scope, methodologies and limitations 2.1 Scope The implementation of this technology adds an extra layer of protection to our infrastructure. This technology is designed to handle validation of HTTP requests only. A great number of attacks happen through HTTP requests. The effectiveness of this system is only as good as the rule sets that are used, and that is where the main problem appears to be. Our challenge is to determine which rules are useful, which are not, and which although may be useful would cause issues such as Denial of Service to clients. Currently the rule set used by zeus-snort are quite generic and some rules apply to particular web servers such as apache or IIS. On the other hand, there are some rules that were not in the original rule set but were found to be quite necessary to include. 2.2 Methodologies Zeus implements HTTP request validation using regular expressions functions from glibc's library thus it is very fast. In our previous tests the performance decrease was negligible with a rule set of 231. 2.3 Limitations As mentioned above the performance decrease is negligible, therefore we can implement very large rule sets or signatures. However, for this first phase we would like to “play safe” and only implement the most recommended rule sets to observe its operation and logs. This is in order to gain knowledge and fine tune the rule sets. 3 Application description 3.1 Functionality The main purpose of this IDS system is to drop “bad” HTTP requests. As mentioned above this is achieved through regular expressions contained in a list which we have named rule set.
